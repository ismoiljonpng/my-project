## 1. О проекте

«AL-ABBOS» — сеть кафе узбекской кухни в Челябинске, **10 точек**. Мы строим **собственный сайт с системой заказа и доставки**, чтобы:

- уйти от зависимости от агрегаторов (Яндекс.Еда забирает большую комиссию);
- решить проблемы старой телефонной доставки — путаница в заказах, неверные адреса, задержки;
- дать каждой точке понятный поток заказа, а курьеру — отдельную панель.

Это рабочий продукт, а не лендинг. Сайт должен выглядеть законченным и красивым «из коробки», а админ должен управлять им **без программиста**.

## 5. Роли пользователей

| Роль | Что может |
|------|-----------|
| **Гость** | Смотреть сайт и меню, собрать корзину. Для оформления заказа — регистрация. |
| **Клиент** (`client`) | Заказывать, видеть свою историю и **отслеживать статус** заказа в реальном времени. |
| **Курьер** (`courier`) | Видит назначенные ему заказы. Принять → забрать с точки → доставить → подтвердить. |
| **Менеджер точки** (`manager`) | Видит заказы своей точки. Принять → готовится → готов к выдаче. *(Опционально; всё это умеет и админ.)* |
| **Админ** (`admin`) | **Полный доступ ко всему** (см. §9). |

Роль хранится в таблице `profiles.role`. По умолчанию при регистрации — `client`. Повышает роли только админ.

## 6. Жизненный цикл заказа (ключевая логика)

Это то, ради чего проект. Статусы заказа (`orders.status`):

| Статус | Кто меняет | Что значит |
|--------|-----------|------------|
| `new` (Новый) | клиент | заказ оформлен, ушёл на выбранную точку |
| `accepted` (Принят) | точка/админ | точка приняла заказ |
| `cooking` (Готовится) | точка/админ | блюда готовятся |
| `ready` (Готов) | точка/админ | готов к выдаче курьеру |
| `courier_assigned` (Назначен курьер) | админ | заказу назначен курьер |
| `picked_up` (Курьер забрал) | курьер | курьер забрал заказ с точки |
| `delivering` (В пути) | курьер | едет к клиенту |
| `delivered` (Доставлен) | курьер | доставлен |
| `confirmed` (Завершён) | клиент/админ | клиент подтвердил получение |
| `cancelled` (Отменён) | админ/точка | заказ отменён (с причиной) |

Правила:
- Клиент при оформлении **выбирает точку** (или система предлагает ближайшую по адресу). Так уходит путаница «с какой точки готовить».
- Каждый статус фиксируется со временем (`updated_at`) — это и есть прозрачность вместо телефонных звонков.
- Статусы движутся только вперёд (кроме `cancelled`). Назад — нельзя.
- Используй **Supabase Realtime**: клиент видит смену статуса без перезагрузки; точка и курьер видят новые заказы сразу.

## 12. База данных

Стартовая схема. Агент дорабатывает и **тестирует** её (Claude Code умеет запускать SQL). Имена — `snake_case`, у всех таблиц `id uuid default gen_random_uuid()` и `created_at timestamptz default now()`.

Таблицы:

- **profiles** — `id` (= `auth.users.id`), `full_name`, `phone`, `role` (`client`/`courier`/`manager`/`admin`), `location_id` (для курьера/менеджера).
- **locations** — `name`, `address`, `lat`, `lng`, `phone`, `work_hours`, `delivery_zone`, `is_active`.
- **menu_categories** — `name`, `slug`, `sort_order`, `image_url`, `is_active`.
- **menu_items** — `category_id`, `name`, `description`, `price`, `image_url`, `is_available`, `is_popular`, `sort_order`.
- **promotions** — `title`, `description`, `image_url`, `discount`, `starts_at`, `ends_at`, `is_active`.
- **banners** — `image_url`, `title`, `link`, `position`, `sort_order`, `is_active`.
- **orders** — `user_id`, `location_id`, `courier_id`, `status`, `total`, `delivery_address`, `lat`, `lng`, `phone`, `comment`, `payment_method`, `updated_at`.
- **order_items** — `order_id`, `menu_item_id`, `name` (снимок), `price` (снимок), `qty`.
- **vacancies** — `title`, `description`, `location_id`, `salary`, `is_active`.
- **job_applications** — `vacancy_id`, `full_name`, `phone`, `message`.
- **site_content** — `key` (уникальный), `value` (jsonb). Для редактируемых текстов/настроек.

Стартовый SQL (доработать и проверить):

```sql
-- Профили + автосоздание при регистрации
create table public.profiles (
  id uuid primary key references auth.users on delete cascade,
  full_name text,
  phone text,
  role text not null default 'client' check (role in ('client','courier','manager','admin')),
  location_id uuid,
  created_at timestamptz default now()
);

create function public.handle_new_user()
returns trigger language plpgsql security definer set search_path = public as $$
begin
  insert into public.profiles (id, full_name)
  values (new.id, new.raw_user_meta_data->>'full_name');
  return new;
end; $$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();

-- Хелпер роли (security definer, чтобы не было рекурсии в RLS)
create function public.user_role()
returns text language sql security definer stable set search_path = public as $$
  select role from public.profiles where id = auth.uid()
$$;
```

В Supabase Storage создай бакеты: `menu` (фото блюд), `promotions` (картинки акций), `banners`. Публичные на чтение, запись — только админ.

## 13. Безопасность и RLS

**RLS включён на каждой таблице.** Логика:

- **Публичное чтение** (для всех, без входа): активные `menu_categories`, `menu_items`, `promotions`, `banners`, `locations`, `vacancies`, нужный `site_content`.
- **profiles**: пользователь читает/меняет свой; админ читает/меняет все.
- **orders**: клиент видит только свои; курьер — только назначенные ему; менеджер — только своей точки; админ — все. Создавать заказ может авторизованный клиент (`auth.uid() = user_id`).
- **Изменение меню, акций, баннеров, точек, вакансий, контента**: только админ (`user_role() = 'admin'`).
- **job_applications**: вставка — для всех (отклик с сайта); чтение — только админ.
- Сервисный ключ Supabase **никогда** не попадает на фронт. Только на сервере (route handlers / server actions), если вообще нужен.

Пример политики:

```sql
alter table public.menu_items enable row level security;

create policy "menu read" on public.menu_items
  for select using (is_available = true or public.user_role() = 'admin');

create policy "menu admin write" on public.menu_items
  for all using (public.user_role() = 'admin') with check (public.user_role() = 'admin');
```

Защити маршруты `/admin`, `/courier`, `/manager` в `middleware.ts`: проверяй роль на сервере, не пускай по чужой ссылке.

## 14. Дизайн и внешний вид

Цель — чтобы **не пришлось ничего передизайнировать**. Делай продуманную дизайн-систему, а не набор разрозненных экранов.

### Дизайн-токены
Все цвета, отступы, радиусы, тени, типографику держи в CSS-переменных + `tailwind.config`. Никаких «магических» значений в компонентах.

### Палитра (тёплый узбекский колорит, но современно и чисто)
- **Акцент / основной**: терракота-оранжевый (восточная глина), напр. `#EA580C`.
- **Второй акцент**: бирюза/изумруд (восточная плитка), напр. `#0D9488`.
- **Светлая тема**: тёплый кремовый фон, тёмно-коричневый текст.
- **Тёмная тема**: глубокий графит/тёмно-коричневый фон, светлый текст, акценты светятся мягче.
- Контраст текста к фону — минимум **4.5:1**.

Задай семантические токены: `--bg`, `--surface`, `--text`, `--text-muted`, `--accent`, `--accent-2`, `--border`, `--radius`. Светлая и тёмная — два набора значений.

### Типографика (важно: шрифты с кириллицей!)
- Заголовки — выразительный шрифт **с поддержкой кириллицы**: `Unbounded` или `Onest`.
- Текст — `Inter` или `Manrope` (тоже кириллица).
- Размеры — через `clamp()` для плавной адаптивности.
- Не используй Arial / системные по умолчанию.

### Компоненты и микровзаимодействия
- Всё на **shadcn/ui**, единый стиль.
- У кнопок — состояния `hover`, `focus` (видимое кольцо), `active` (нажатие), `disabled`, `loading`.
- Иконки — **lucide-react**, к месту (в навигации, кнопках, статусах, карточках).
- Карточки блюд — фото, аккуратная тень, лёгкий hover-подъём.
- Скелетоны при загрузке, аккуратные пустые состояния, понятные ошибки.

### Адаптив и доступность
- **Mobile-first.** Проверяй мысленно на 375 / 768 / 1280 px.
- У всех картинок — `alt`. Навигация с клавиатуры. ARIA где нужно.

## 15. Контент (реальный, не lorem)

Заполни сайт настоящими данными в `seed.sql`.

**Блюда (примерный набор, дополни):** плов (ферганский, самаркандский), лагман, шурпа, мастава, манты, чучвара, самса (с мясом, с тыквой), шашлык (баранина, курица, люля), казан-кебаб, димляма, нарын, ханум, салат «Ачичук», морковча, чай (чёрный, зелёный), айран, компот. Реальные цены в рублях, короткие аппетитные описания.

**Точки:** 10 точек по Челябинску — правдоподобные адреса, телефоны, часы, координаты. *(Владелец потом заменит на настоящие.)*

**Акции (пример):** «Два плова по цене одного по будням до 16:00», «Бесплатная доставка от 1500 ₽», «−15% на первый заказ через сайт».

**О нас:** живой текст про сеть узбекской кухни, гостеприимство, почему запустили свою доставку.

**Вакансии (пример):** повар, кассир, курьер, помощник на кухню — с точкой и зарплатой.

**Фото:** на старте — ссылки на фото еды (например, Unsplash) или аккуратные плейсхолдеры. В коде/README отметь, что админ загрузит реальные фото через Storage.

## 16. Анимации и фоны

Сайт должен «жить», но не раздражать и не тормозить.

- **Hero на главной** — один выразительный анимированный фон: мягкие движущиеся градиенты / «блобы», или лёгкие частицы на Canvas, или медленный Three.js. Узнаваемый, но не кричащий.
- **Секции** — разные, но **гармоничные** фоны: тонкие градиенты, мягкие пятна, ненавязчивый восточный паттерн. Разнообразие в рамках одной палитры, а не «зоопарк».
- **Появление при скролле** — Framer Motion (fade + лёгкий подъём), с easing.
- **Анимации**: минимум 300 мс, всегда easing (`power2.out`, `ease`), **никаких линейных** и резких скачков. Зацикленные — без видимого «прыжка».
- **Производительность**: `requestAnimationFrame`, на мобильном — меньше частиц/проще геометрия, `devicePixelRatio` не выше 2, чистка ресурсов при размонтировании.
- **Обязательно уважай `prefers-reduced-motion`** — для таких пользователей отключай тяжёлые анимации:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after { animation-duration: .01ms !important; transition-duration: .01ms !important; }
}
```

## 17. Светлая / тёмная тема

- Через **next-themes**, переключатель в шапке (солнце/луна), по умолчанию — системная.
- Без «мигания» при загрузке (корректная инициализация темы).
- Оба варианта должны быть одинаково красивыми и читаемыми — тёмная тема не «инверсия наугад», а отдельный продуманный набор токенов.

## 18. Переменные окружения

`.env.example` (закоммитить с плейсхолдерами; реальный `.env.local` — в `.gitignore`):

```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
NEXT_PUBLIC_YANDEX_MAPS_API_KEY=your-yandex-maps-key
# Серверный ключ — только если реально нужен на сервере, на фронт не выводить:
# SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
```

В `.gitignore`: `.env`, `.env.local`, `.env.*.local`. Те же переменные добавить в **Vercel → Settings → Environment Variables**.

## 19. Порядок запуска (для владельца)

1. **GitHub** — создать репозиторий, залить код.
2. **Supabase** — создать проект → в SQL Editor прогнать `schema.sql`, `policies.sql`, `seed.sql` → в Storage создать бакеты `menu`, `promotions`, `banners`.
3. Скопировать `URL` и `anon key` из Supabase в `.env.local`.
4. Получить ключ **Яндекс.Карт** (Кабинет разработчика) → в `.env.local`.
5. `npm install` → `npm run dev` → открыть `http://localhost:3000`.
6. Зарегистрироваться на сайте, затем **сделать себя админом** (см. §20).
7. **Vercel** — подключить репозиторий, добавить те же переменные окружения, задеплоить.
8. В Supabase Auth прописать домен с Vercel в Redirect URLs (для подтверждения почты и входа).

После деплоя дай владельцу: ссылку на сайт, как зайти в `/admin`, как загрузить реальные фото и заменить адреса точек.

## 20. Первый администратор

Роль `admin` через интерфейс никто выдать не может (это и есть защита). Поэтому первого админа назначаем вручную в Supabase:

```sql
update public.profiles set role = 'admin'
where id = (select id from auth.users where email = 'почта_владельца@example.com');
```

Дальше владелец-админ назначает остальные роли (курьеров, менеджеров) уже из админки.

## 22. План по этапам (так и собирай)

1. **Каркас**: Next.js + Tailwind + shadcn/ui + next-themes, шапка/подвал, переключатель темы, дизайн-токены, шрифты.
2. **База данных**: схема + RLS + seed, бакеты Storage, клиенты Supabase (`client`/`server`/middleware).
3. **Публичный сайт**: Главная, Меню, Акции, Контакты (с картой), О нас, Вакансии — на реальных данных.
4. **Корзина и заказ**: корзина → оформление → запись в `orders`/`order_items`.
5. **Авторизация и кабинет клиента**: регистрация/вход, история, отслеживание статуса (Realtime).
6. **Админ-панель**: заказы, меню, акции, баннеры, точки, роли, вакансии, контент (+ аналитика).
7. **Панель курьера** (и при желании менеджера точки).
8. **Полировка**: анимации, фоны, адаптив, доступность, пустые/ошибочные состояния.
9. **Деплой**: Vercel + переменные окружения + проверка прода.

После каждой фазы — рабочее состояние и короткий отчёт владельцу простым языком.

## 23. Чего НЕ делать

- ❌ Заглушки, `lorem ipsum`, `TODO`, пустые страницы в финале.
- ❌ Захардкоженный контент, который нельзя поменять из админки.
- ❌ Ключи и секреты в коде или в репозитории.
- ❌ Таблицы без RLS.
- ❌ Линейные/резкие анимации и тяжёлые эффекты без учёта `prefers-reduced-motion` и мобильных.
- ❌ Удалять данные, сбрасывать схему или перезаписывать файлы без подтверждения владельца.
- ❌ Английский в пользовательском интерфейсе.
